## 执行步骤

**1.1 检查 S1 完成状态**

读取 `session.json`，确认 `stages.S1.status === 'completed'`。若未完成：

```
❌ 无法进入 S2 技术方案阶段
   S1 PRD 生成尚未完成（当前状态: {s1Status}）
   请先完成 S1 阶段后再进入 S2。
```

**1.2 加载 S1 产出物**

读取 `{session_dir}/01-prd.md`，若文件缺失则阻止执行。

**1.3 更新 Session 状态**

```
S2 → in_progress
```

**1.4 项目工程路径**

默认为当前工作区根目录 `.`，无需询问用户。

### Step 3: 技术方案生成

**必含章节**：

| 章节 | 说明 |
|------|------|
| 概述 | 需求摘要、技术选型、项目类型（全新/增量） |
| 页面结构设计 | 页面列表（含变更类型 🆕/✏️/♻️）、路由配置、导航关系图（Mermaid） |
| 组件拆分 | 组件树结构、自定义组件定义（props/events）、内置组件使用清单 |
| 数据模型定义 | 页面/组件数据模型（精确到字段名、类型、默认值）、共享数据结构 |
| API 调用方案 | 系统 API 清单（含已声明/需新增标注）、调用时机、参数、错误处理策略 |
| 状态管理方案 | 页面内状态、跨页面状态、数据流向图（Mermaid） |
| 文件目录结构 | 完整项目目录（含变更类型标注） |
| 变更影响分析 | 仅已有项目：manifest.json 变更、已有文件修改清单、新增文件清单 |
| 技术风险与应对 | 风险点和解决方案 |

**写作规范**：
- 面向编码 Agent 写作：必须包含足够实现细节，禁止"待定"等占位描述
- 已有项目采用增量设计，标注每个文件的变更类型
- 数据模型精确到字段级别
- API 引用必须从知识库中查找，确保签名、参数准确
- 始终使用简体中文

### Step 5: Checkpoint 交互

展示摘要：

```
📋 技术方案摘要:
   • 需求名称: {requirement_name}
   • 页面数: {page_count} 个
   • 自定义组件数: {component_count} 个
   • 系统 API 调用: {api_count} 个
   • 数据模型数: {model_count} 个
   • 文件目录层级: {dir_depth} 层
```

操作选项：

```
❓ 请选择操作:
   [y] 确认 — 保存技术方案，标记 S2 完成，进入 S3 功能研发
   [e] 编辑 — 提供修改意见，迭代生成
   [n] 放弃 — 丢弃当前技术方案，重新生成

⏳ 等待您的输入...
```

**必须阻塞等待用户输入，严禁自动进入 S3。**

| 命令 | 处理逻辑 |
|------|---------|
| `y` | 标记 S2 为 `completed`，流转到 S3 |
| `e` | 接收修改意见，追加上下文，重新生成（支持多轮迭代） |
| `n` | 丢弃产出，重置为 `in_progress`，重新执行 |
